현재 작업을 마무리한다. CLAUDE.md §15 협업·Git 컨벤션을 그대로 따른다.

## 순서

1. **현재 브랜치 확인**: `git branch --show-current` → 브랜치명에서 type과 이슈 번호 파싱.
   - 예: `feat/#12-login-api` → type=`feat`, 이슈번호=12
   - `main`/`master`/`dev` 브랜치면 즉시 중단: "작업 브랜치가 아닙니다. `/task-start` 먼저 실행하세요."
   - 형식이 안 맞으면 사용자에게 이슈 번호를 묻고 진행.

2. **변경사항 파악** (병렬 실행):
   - `git status`
   - `git diff` (스테이징/언스테이징 모두)
   - `git log main..HEAD` (현 브랜치의 커밋 이력)

3. **커밋 메시지 작성**: 변경 파일 리뷰해서 작성. 형식: `<type>: <한국어 설명> (#<이슈번호>)`.
   - 예: `feat: 이메일 유효성 검사 추가 (#12)`
   - 변경의 "why"에 초점, 1~2 문장.

4. **Staging**: 변경 파일을 명시적으로 add (`git add <파일1> <파일2>`). `git add -A`/`-u`/`.` **금지** (`.env`, credentials 등 사고 방지).

5. **커밋 생성**: HEREDOC + `Co-Authored-By` 포함.
   ```
   git commit -m "$(cat <<'EOF'
   <type>: <설명> (#N)

   Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
   EOF
   )"
   ```

6. **푸시**: `git push -u origin <현재브랜치>` 실행. 훅 실패 시 NEW 커밋으로 보강 (amend 금지).

7. **PR 본문 작성**: 아래 "PR 본문 작성 가이드" 따라 마크다운 본문 작성.

8. **PR 생성**: `gh pr create --base main --title "<커밋제목>" --body "<위본문>"` 실행. HEREDOC으로 body 전달.

9. **결과 보고**: PR URL 한 줄.

> 자동 머지는 하지 않는다. 리뷰 1명 승인 후 사용자가 직접 Squash and Merge (CLAUDE.md §15.4).

## PR 본문 작성 가이드

**대상 독자**: 리뷰어가 PR 페이지 열었을 때 **30초 안에 "왜 이걸 머지해도 되는지"** 판단 가능해야 한다. 코드 디테일은 diff 가 보여주니, 본문은 **맥락과 의도**에 집중.

### 톤 원칙

- 첫 섹션은 비전공자도 이해 가능한 한국어로. 약어/용어는 처음 쓸 때 풀어서.
- "왜" 가 "무엇" 보다 먼저. "무엇" 은 코드/diff 가 이미 보여준다.
- 한 문장에 한 가지 사실. 짧게.
- SQL/코드 인라인은 **그 결정을 정당화하는 핵심 한두 줄만**. 전체 복붙 금지.
- 표·헤더 남발 금지. 4개 섹션 (왜 / 변경 / 검증 / 제외) 이면 충분.

### 본문 구조

```markdown
## 왜

<1~3 문장. 어떤 문제/요청/기회가 있었고, 이 PR 이 어떻게 해결하는지.
이슈 본문과 중복돼도 OK — 리뷰어가 이슈를 안 열어도 맥락 잡혀야 함.>

## 변경점

- <변경 1 — 1줄. 의도 중심.>
- <변경 2 — 1줄.>
- <변경 3 — 1줄.>

(설계 결정 근거가 필요한 항목만 1~2 문장 부연. 항목당 5줄 넘으면 분리.)

## 확인 (Test plan)

- [ ] <자동: CI / 새 IT 통과>
- [ ] <수동: 어떤 시나리오를 어디서 클릭/요청 했을 때 어떤 결과>
- [ ] <외부 의존: Supabase 환경, PG webhook 등>

## 이번에 안 하는 것 (선택)

- <스코프에서 빠진 것 — 별도 이슈/PR 로 가는지 명시>

Closes #<이슈번호>
```

### 길이 가이드

- 본문 전체 **600~900자** 권장. 그 이상은 변경이 너무 큰 건 아닌지 한 번 더.
- "변경점" 5개 넘어가면 PR 을 쪼갤 수 있는지 검토.
- 깊은 설계 배경은 PR 본문 대신 코드 주석 / 이슈 댓글로 분산.

### 좋은 예 / 나쁜 예

| 나쁜 예 (기술 dump) | 좋은 예 (의도 중심) |
|---|---|
| "DO $$ ... $$ 가드 + ON CONFLICT DO NOTHING idempotent + SECURITY DEFINER + search_path 고정" | "Supabase 가입 시 우리 DB 에 사용자 자동 등록. 로컬 테스트 환경에선 동작 안 하도록 안전장치 포함." |
| Summary 에 SQL 30줄 인라인 | "전체 SQL 은 V3 마이그레이션 파일 참고. 핵심 결정: nickname 충돌 0% 보장 위해 UUID 전체 사용." |
| "8자만 쓰면 √(16^8)≈65k 에서 birthday collision" | "닉네임 8자만 쓰면 ~65,000 가입자 시점부터 중복 위험. 전체 UUID 쓰면 절대 안 겹침." |
