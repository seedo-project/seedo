새 작업을 시작한다. 인자: $ARGUMENTS

CLAUDE.md §15 협업·Git 컨벤션을 그대로 따른다.

## 순서

1. **type 검증**: 인자에서 첫 토큰을 `type`으로 추출. 허용값: `feat` / `fix` / `docs` / `refactor` / `chore` / `style` / `test`. 잘못되었으면 에러로 중단하고 사용법 안내 (`/task-start feat "로그인 API 구현"`).

2. **이슈 제목 작성**: 나머지 인자를 `설명`으로 사용. type을 PascalCase로 변환 → 제목 형식 `[<TypePascal>] <설명>`.
   - 예: `feat` + `로그인 API 구현` → `[Feat] 로그인 API 구현`

3. **라벨 매핑**:
   - `feat` → `feature`
   - `fix` → `bug`
   - `docs` → `documentation`
   - `refactor` → `refactor`
   - `chore` → `chore`
   - `style` → `style`
   - `test` → `test`

4. **이슈 본문 작성**: 아래 "본문 작성 가이드" 따라 마크다운 본문 작성.

5. **이슈 생성**: `gh issue create --title "<제목>" --label <라벨> --body "<본문>"` 실행 (HEREDOC). 라벨이 없다는 에러가 나면 라벨 없이 재시도. 출력에서 이슈 URL과 번호 파싱.

6. **슬러그 생성**: 설명을 영문 케밥-케이스로 변환 (한글이면 적절히 영문 의미로). 5단어 이내. 예: `로그인 API 구현` → `login-api`.

7. **브랜치 생성**: 현재 브랜치가 `main` 인지 확인 후 (아니면 사용자에게 한 번 확인) `git checkout -b <type>/#<이슈번호>-<슬러그>` 실행.
   - 예: `feat/#12-login-api`

8. **결과 보고**: 이슈 URL + 새 브랜치명 + "이제 작업 시작하세요" 한 줄.

이후 일반 대화·구현 모드로 복귀한다.

## 본문 작성 가이드

**대상 독자**: 동료 개발자뿐 아니라 PM·디자이너도 첫 두 문단만 읽고 "왜 필요한지" 알 수 있어야 한다. 기술문서가 아니라 **팀 공유용 메모** 톤으로.

### 톤 원칙

- 첫 문단은 **그 도메인을 모르는 사람도 이해 가능**하게 — 풀어서 설명, 약어/용어는 처음 쓸 때 한 번 풀어 쓰기 (예: "RBAC(역할 기반 권한)").
- 두 번째 문단부터 기술 디테일 들어가도 됨.
- 짧은 문장. 한 문장에 한 가지 사실.
- "왜" 가 "무엇" 보다 먼저.
- 표/긴 코드블록은 최소화 — 핵심만. 전체 SQL/스키마 붙여넣기 금지 (PR 의 diff 가 더 정확함).

### 본문 구조

```markdown
## 왜 필요한가

<1~3 문장. 비전공자도 이해할 한국어. 어떤 문제/불편/기회가 있고, 이 작업이 어떻게 해결하는지.>

## 무엇을 한다

- <변경점 1 — 1줄>
- <변경점 2 — 1줄>
- <변경점 3 — 1줄>

(필요하면 각 항목 아래에 짧은 설명 1~2줄. 깊은 디테일은 PR 에서.)

## 검증 방법

- <어떻게 동작 확인할지 — 자동 테스트 / 수동 확인 / 외부 의존>

## 이번에 안 하는 것 (선택)

- <스코프에서 빠진 것 + 왜 (별도 이슈로 빠질지 등)>

Closes 이 이슈
```

### 길이 가이드

- 본문 전체 **400~700자** 권장. 그 이상은 잘게 쪼갤 수 있는지 검토.
- 한 섹션 5줄 넘으면 정말 다 필요한지 한 번 더.

### 좋은 예 / 나쁜 예

| 나쁜 예 (기술 dump) | 좋은 예 (사람 친화) |
|---|---|
| "RBAC 조회 시 unknown user 로 권한 빈 list → 403" | "회원가입 직후 마이페이지나 글쓰기 시도하면 권한이 없다고 막힘. 새 가입자가 우리 DB 에 자동으로 등록되지 않기 때문." |
| "SECURITY DEFINER + search_path schema-shadowing 방지" | "함수 권한 격리 (Supabase 권장 패턴)" |
| "CHECK ((status = 'DELETED') = (deleted_at IS NOT NULL))" 전체 SQL 붙여넣기 | "삭제된 사용자 행은 deleted_at 이 채워져 있어야 한다는 제약 추가" |
